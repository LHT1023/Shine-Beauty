## 一、30 秒电梯演讲（必须背熟）

> **场景**：面试官说"给我介绍一下你的项目"

**中文版：**
"Shine Beauty 是我独立设计并开发的一款 AI 粉底推荐工具。核心问题是：彩妆市场产品信息过载，消费者很难在 60+ 款粉底里快速找到适合自己的。我设计了一套三层推荐架构——关键词快速匹配层、LLM 意图理解层、兜底热门推荐层——来平衡响应速度、推荐精准度和成本三个维度。我还加了埋点系统，模拟了 12 个用户行为场景进行验证，通过率 100%。这个项目让我实践了 AI 产品从问题定义、架构设计、指标设定到测试验证的完整 PM 闭环。"

**关键数字要记住：**
- 产品库：60+ 款粉底

- 硬过滤保证：推荐产品 100% 符合用户条件，零幻觉
- 埋点指标：推荐层分布、推荐数量、查询语言、响应时间 P95

## 二、问题定义（Problem Framing）

### 背景
- 美妆市场产品过载，Sephora 单粉底品类 200+ SKU
- 用户缺乏个性化引导，依赖 KOL/口碑/试用
- 线上购买转化率低的核心原因之一：不知道哪款适合自己

### 你解决的核心问题
**"信息不对称导致的决策摩擦"**
- 用户知道自己的诉求（油皮、哑光、100 以内），但不知道哪款产品满足
- 现有解决方案：搜索引擎（太泛）、KOL 推荐（不个性化）、品牌导购（有利益立场）

### 为什么选 AI
- 用户偏好是多维度的（肤质 × 妆效 × 预算 × 关注点），规则引擎组合爆炸
- 自然语言交互门槛最低，用户不需要学习如何使用
- LLM 能理解模糊表达（"水水的感觉"、"像没上妆"）

## 三、用户画像（User Personas）

### Persona 1：小红书用户「小仪」
- **背景**：22 岁大学生，关注美妆 UP 主，预算有限（<200 RMB）
- **行为**：常用中文查询，喜欢问"油皮哑光推荐"
- **痛点**：推荐太多，不知道哪款真的适合自己肤质
- **对产品的需求**：快速、精准、有理由地给出推荐
- **对应系统设计**：CN_EN_MAP 归一化层、预算硬过滤、Layer 1 <50ms 命中

### Persona 2：美妆爱好者「Sophie」
- **背景**：25 岁职场人，英文用户，品牌意识强
- **行为**：会问"NARS foundation for oily skin"，有品牌倾向
- **痛点**：知道品牌但不确定具体哪款
- **对产品的需求**：品牌+肤质精准匹配，响应快
- **对应系统设计**：品牌名进入硬过滤，Layer 1 直接命中，41ms 返回

### Persona 3：初学者「小敏」
- **背景**：18 岁第一次买粉底，完全不懂
- **行为**：发"推荐一个粉底给我"这种模糊消息
- **痛点**：不知道自己需要什么，容易被推销
- **对产品的需求**：被引导，而不是被推送
- **对应系统设计**：意图不完整时触发精准追问，不盲目推荐

## 四、技术架构深度讲解（项目核心）

> 这一节是面试时讲"技术背后的产品思考"的核心材料

### 整体架构

```
用户输入（中/英文自然语言）
        ↓
    Layer 0: normalize()  ← 语言归一化（中文→英文映射）
        ↓
    isContextualQuery?    ← 引用上文检测（"那个"/"第一个"）
    ↙ Yes      ↘ No
   L2          isKeywordMatchable? + mergedMessage（跨轮合并）
               ↙ Yes               ↘ No
              L1                    L2
        ↓
    Layer 1: keywordMatch()
      硬过滤（全部条件必须满足）→ 按产品评分排序 → Top 3
        ↓ 0结果
    Layer 2a: extractIntent()  ← LLM 只做翻译，不做推荐
        ↓
      isComplete? → No → buildFollowUpFromIntent()（精准追问）
        ↓ Yes
    Layer 2b: intentMatch()
      MongoDB硬过滤 → 软评分 → Top 3
        ↓ 0结果
    Layer 3: fallbackRecommendation()  ← 评分最高 Top 3，永不崩溃
        ↓
    Analytics Middleware（异步埋点，不阻塞响应）
        ↓
    返回给前端 { message, recommendations, _meta }
```

---

### Layer 0：语言归一化 `normalize()`

**做什么：** 把 30 个中文美妆词通过字典映射成英文，统一进入同一套推荐引擎。

```
"油皮哑光粉底液" → "oily matte foundation"
"200以内"        → "under 200"
"敏感肌水光"     → "sensitive dewy"
```

**PM 视角（必须会说）：**
> "这是一个体验公平性问题。如果没有这一层，中文用户'哑光'识别不了，查询会降级到 LLM 层，响应时间从 40ms 变成 2s，是英文用户的 50 倍。这是隐性的用户歧视，修复优先级应该高于新功能。"

---

### Layer 1：关键词引擎 `keywordMatch()`

**核心设计哲学：用户说的 = 硬条件，用户没说的 = 不干涉**

**Step 1：硬过滤 vs 软偏好识别**

用 `SOFT_SIGNALS` 正则检测关键词前 40 个字符有没有修饰语：

```
"I have oily skin"            → hard.skinType = 'oily'  （必须满足）
"prefer matte if possible"    → soft.finish = 'matte'   （满足加分，不满足不排除）
"must be vegan"               → hard.vegan = true        （必须满足）
"ideally under $40"           → soft.budget = 40         （满足加分）
```

软偏好关键词：prefer / if possible / ideally / would love / 最好 / 如果可以 / 希望

**Step 2：硬过滤（链式 .filter()）**

```
全库 60 产品
  → 过滤 skinType=oily  → 剩 24 款
  → 过滤 finish=matte   → 剩 11 款
  → 过滤 price≤30       → 剩 6 款
  → 结果为 0 → 直接返回空，进 Layer 2
```

关键：每个硬条件是独立的筛子，全部通过才留下，一个不满足就排除。

**Step 3：软评分排序**

基础分 = 产品真实 rating（4.0–4.8），软偏好匹配 +0.5，concern 匹配 +0.3。

**为什么用产品 rating 而不是自己定权重？**
> "用我们自己定的权重，本质上是在替用户决定哪个维度更重要。但用户才是最了解自己需求的人。产品自身的评分是市场上真实用户验证过的质量信号，比我们主观判断更客观。"

**进 Layer 1 的两道门**

1. `isContextualQuery(message)`：检测引用信号词（"that one / 第一个 / alternative / cheaper version"）。命中直接跳到 Layer 2，因为这类问题依赖上下文，关键词引擎理解不了。

2. `isKeywordMatchable(mergedMessage)`：检测是否有 ≥2 个决策维度。维度包括：肤质、妆效、预算、遮瑕度、品牌。维度不够就不进 Layer 1，避免返回过于宽泛的结果。

**多轮跨轮合并（重要）**

```javascript
// 合并最近 2 轮用户消息，Layer 1 能跨轮理解意图
const mergedMessage = [...recentUserTexts, message].join(' ');
```

效果：
```
第1轮："我是油皮"（单维度，Layer 2 追问）
第2轮："要哑光的"
  → mergedMessage = "我是油皮 要哑光的"
  → 检测到 oily + matte（双维度）
  → Layer 1 命中，返回适合油皮的哑光粉底 3 款，全部 100% 符合两个条件
```

---

### Layer 2a：LLM 意图提取 `extractIntent()`

**LLM 在这里只做一件事：自然语言 → 结构化 JSON**

Prompt 约 150 tokens，输出：

```json
{
  "skinType": "oily",
  "finish": "matte",
  "coverage": null,
  "budgetMax": 40,
  "brands": [],
  "concerns": ["acne"],
  "vegan": null,
  "crueltyFree": null,
  "spf": null,
  "isComplete": true
}
```

LLM 看不到任何产品数据，所以推荐结果里的每一款产品都来自真实数据库，不可能凭空捏造。

**两道防护栏：**

1. **枚举白名单校验**：LLM 输出 "combination-oily" 这种非法值 → 直接置 null，不进数据库查询
2. **isComplete 服务端重算**：不信任 LLM 的判断，自己算 `有效字段数 ≥ 2 || 有布尔属性(vegan/spf/crueltyFree)`

**多轮上下文传递：**

context 里包含最近 6 轮对话 + 每轮推荐的产品名：

```
User: I have oily skin
Shine: Got it — could you share your finish preference? [Recommended: ]
User: matte please
Shine: Here are my picks [Recommended: Double Wear, MAC Studio Fix, NARS]
User: what about the first one, is there a cheaper version?
  → LLM 从 context 里读到 Double Wear，理解"第一个"是它
```

---

### Layer 2b：自有匹配引擎 `intentMatch()`

和 Layer 1 的 keywordMatch 相比，多了一个容错机制：

```javascript
// 硬过滤结果 < 2 时，放宽肤质限制（保留其他约束）
if (products.length < 2 && intent.skinType) {
    delete relaxed.skinTypes;
    // 重新查询
}
```

用户要 sensitive skin + full coverage + $20，库里可能没有，自动放宽肤质，保住预算和遮瑕度约束，避免完全返回空结果。

---

### Layer 3：兜底 `fallbackRecommendation()`

直接查 `Foundation.find({}).sort({ rating: -1 }).limit(3)`，永不崩溃。

触发条件：超低预算（$5以内）、条件极端冲突（dry skin + 强控油）等极端情况。

测试中 Layer 3 触发率 8%（12 个场景中 1 个），属于正常的安全网。

---

### 幻觉（Hallucination）是如何被解决的

**旧架构（有幻觉）：**
把 60 个产品全塞进 prompt，GPT 可能推荐数据库里不存在的产品，或编造属性。

**新架构（幻觉封死）：**

| 幻觉类型 | 防护手段 |
|---|---|
| 推荐不存在的产品 | LLM 看不到产品数据，所有产品来自 MongoDB 查询 |
| 伪造产品属性 | 推荐理由由引擎生成，直接读数据库字段 |
| 非法枚举值进查询 | VALID_ENUMS 白名单校验，非法值置 null |
| concerns 自由文本偏差 | 只影响软评分小幅加减，不影响哪些产品出现 |

---

### Analytics 埋点系统

采用**拦截 `res.json`** 的方式，无侵入接入所有路由：

```javascript
res.json = function(body) {
  setImmediate(async () => {          // 异步，不阻塞主响应
    await Analytics.create(event);   // 写入 MongoDB
  });
  return originalJson.call(this, body);
}
```

记录的事件：

| 事件类型 | 字段 | 用途 |
|---|---|---|
| chat_query | query, queryLanguage, aiLayer, aiMethod, recommendationCount | 推荐效果分析 |
| browse_products | filters, resultCount | 用户浏览行为 |
| favorite_add/remove | userId, productId | 推荐接受信号 |

语言检测：中文字符占比 > 50% → zh，> 10% → mixed，否则 en。

## 六、产品设计决策（核心 PM 内容）

### 决策 1：为什么用三层架构，而不是纯 LLM？

**PM 思路**（用成本/速度/质量三角来讲）：
- **成本**：LLM API 调用有费用，高频场景下纯 LLM 不可持续
- **速度**：LLM 响应 1-3s，Layer 1 关键词 37-52ms，差距 50 倍
- **质量**：对于明确意图，规则引擎比 LLM 更可靠（不会幻觉）

**决策原则：能用规则解决的不用 AI，AI 用在规则解决不了的地方**

实际效果：Layer 1 命中 50% 的请求，这 50% 的成本趋近于零。

### 决策 2：为什么 LLM 只做意图提取，不做推荐？

**原设计**：把 60 个产品全部传给 GPT，让它直接给出推荐
**问题**：
1. Prompt 过长（60 产品 × ~200 token = 约 12,000 token），成本高
2. 推荐逻辑在黑盒里，无法解释、无法调优
3. LLM 幻觉风险，可能推荐不存在的产品或编造属性

**新设计**：LLM 只提取 9 个字段（skinType / finish / coverage / budgetMax / brands / concerns / vegan / crueltyFree / spf）
- Token 成本下降 ~80%（prompt 从 ~12,000 token 降到 ~150 token）
- 推荐逻辑完全可解释（每条推荐有 reason 字段）
- 幻觉封死（产品来自 MongoDB 真实数据）

**面试话术：**
> "这个决策的本质是重新划分 AI 和确定性系统的边界。AI 做它最擅长的——自然语言理解；确定性系统做它最擅长的——精确过滤和排序。这样既保留了 AI 的灵活性，又保证了推荐的可解释性和可控性。"

### 决策 3：为什么用户说的所有条件都应该变成硬过滤？

**旧设计（加权打分）**：给每个维度定权重（肤质+3分、妆效+3分、预算+4分），高分产品胜出。

**问题**：权重是我们主观定的，用户强调"一定要在$30以内"，但预算在打分体系里可能排名不如"油皮适配性"，出现超预算产品排在前面的情况。

**新设计（硬过滤+产品评分排序）**：
- 用户明确说的条件 → 全部进硬过滤，0 容忍
- 带 prefer/最好 修饰 → 软偏好，满足加分但不强制
- 过滤后排序 → 产品自身评分（market-validated quality signal）

**PM 视角：**
> "这是一个用户优先还是系统优先的选择。加权打分本质上是系统在替用户决定哪个维度重要，但用户才最了解自己。用硬过滤把控制权还给用户——你说的条件就是不可违反的约束，系统只在你的约束范围内找最好的。"

### 决策 4：追问机制 — Precision vs Friction

**核心 tradeoff**：追问越多，推荐越精准；但追问越多，用户摩擦越大，可能放弃对话。

**判断标准**：只在信息不足会导致推荐完全不准时才追问。

- 单维度（只说"我是油皮"）→ 追问：60 款里有 40 款适合油皮，推出去没意义
- 双维度（"油皮+哑光"）→ 直接推：Layer 1 硬过滤后剩合适的产品，全部符合条件

**isComplete 计算逻辑（服务端，不依赖 LLM）：**
```
有效字段数 ≥ 2（skinType + finish / skinType + budget 等组合）
或者有明确 boolean 属性（vegan=true / crueltyFree=true / spf=true）
→ isComplete = true，直接推荐
```

### 决策 5：多轮对话上下文如何管理？

**三层设计：**

1. **Layer 1 跨轮合并**：合并最近 2 轮用户消息做关键词提取，解决"分轮补充信息"场景
2. **Layer 2 LLM 读历史**：最近 6 轮对话 + 推荐产品名都放进 context，LLM 能理解指代
3. **contextualQuery 路由**：检测"that one / 第一个 / cheaper version"等引用信号，自动跳过 Layer 1

**为什么不把所有查询都走 Layer 2？**
> "因为 Layer 2 要等 LLM 响应，最快也要 1.2s。对于明确查询（油皮+哑光+$30），Layer 1 43ms 就能给出精准结果，没必要让用户等 30 倍的时间。只有真正需要上下文理解的查询才路由到 Layer 2，这是资源分配的精细化。"

## 七、数据与指标（Metrics）

### 核心 KPI 框架（HEART 框架）

| 维度 | 指标 | 为什么重要 |
|------|------|-----------|
| **Happiness** | 用户是否接受推荐（收藏率） | 推荐质量的直接信号 |
| **Engagement** | 平均对话轮数 | 用户是否在深度使用 |
| **Adoption** | 首次使用到第二次使用的转化率 | 产品是否创造了真实价值 |
| **Retention** | 7 日回访率 | 长期价值 |
| **Task Success** | 推荐成功率（有推荐 vs 只有追问） | 系统能力指标 |

### 你已埋的指标
- `chat_query`：推荐层（L1/L2/L3）、推荐数、查询语言（zh/en/mixed）、响应时间、提取的 intent
- `browse_products`：筛选条件、结果数
- `favorite_add/remove`：用户接受推荐的信号

### 你没埋但应该埋的（面试可以主动讲）
- **推荐接受率**：用户看到推荐后是否点击/收藏（North Star Metric）
- **追问完成率**：被追问的用户是否继续对话还是直接离开
- **session 深度**：每次会话的对话轮数
- **语言切换率**：中文用户是否因体验差切换成英文

### North Star Metric：推荐接受率

> 不能只看"返回了推荐"，这是系统指标，不是用户指标。
>
> **真正的推荐成功** = 用户看到推荐 → 点击查看详情 or 加入收藏 or 回访时仍在使用同款
>
> **代理指标（没有真实用户时）**：推荐产品是否 100% 匹配用户声明的硬条件（已验证）

## 八、测试与验证

### 如何在没有真实用户时验证产品

设计了 12 个**用户行为模拟场景**，覆盖三类：
1. **Good Cases（6个）**：正常用户给出明确需求，验证推荐准确性
2. **Multi-turn（1个）**：验证追问→补充信息→推荐的完整多轮流程
3. **Edge Cases（5个）**：模糊、跑题、条件冲突、超低预算，验证鲁棒性

### 验证的维度不只是"有没有推荐"

每个场景的 evaluate() 函数验证：
- 推荐产品是否符合用户明确条件（$30预算内的产品价格是否 ≤30）
- 油皮查询返回的产品是否有 skinTypes.includes('oily')
- Vegan 查询返回的产品是否有 isVegan=true
- 追问是否精准（告知用户具体缺了哪个维度）

### 面试讲法

> "没有真实用户，我就设计了模拟测试来替代用户调研。每个场景相当于一个用户故事（User Story），我不只验证系统有没有返回推荐，还验证推荐的每一款产品是否真的满足用户声明的条件——这才是真正的精准度验证。通过测试发现了推荐引擎的系统性问题，定位根因后修复，推荐准确率从部分不达标迭代到 100% 硬条件符合率。"

## 九、迭代规划

### 优先级排序（RICE 框架）

| 功能 | Reach | Impact | Confidence | Effort | 优先级 |
|------|-------|--------|------------|--------|--------|
| 用户反馈机制（推荐好/不好） | 高 | 高 | 高 | 低 | ⭐⭐⭐ 最高 |
| 推荐接受率埋点 | 高 | 高 | 高 | 极低 | ⭐⭐⭐ 最高 |
| vegan/crueltyFree 进 Layer 1 检测 | 中 | 中 | 高 | 低 | ⭐⭐ |
| 跨会话用户偏好记忆 | 高 | 高 | 中 | 高 | ⭐⭐ |
| 图片输入（上传肤色识别） | 低 | 高 | 低 | 高 | ⭐ |

### 最高优先：用户反馈机制

> "现在我知道系统推荐了什么，但不知道用户是否喜欢。这是一个数据盲区。加一个简单的'推荐帮助到你了吗？'按钮，成本极低，但能建立真正的推荐质量反馈循环，为后续优化提供 Ground Truth。"

## 十、面试高频问题 Q&A

### 📌 项目类问题

**Q1：你为什么做这个项目？**

> "我在准备 PM 面试时意识到，只讲理论不够——我需要一个能展示完整产品思维的作品。我选择美妆推荐这个方向，是因为它涉及个性化推荐、自然语言交互、AI 产品设计三个我想深入的领域。更重要的是，我遇到了这个真实问题：Sephora 上 200+ 款粉底，没有任何工具能根据我的肤质和预算快速筛选。这就是我的 Product Opportunity。"

---

**Q2：项目中最难的决策是什么？**

> "最难的决策是：LLM 应该做什么，不应该做什么。最初我让 LLM 直接做推荐，结果发现三个问题：成本高（12,000 token/次）、不可解释、有幻觉风险。但如果完全不用 LLM，系统处理不了自然语言的模糊性。
>
> 最终决策：用 LLM 做 NLU（意图提取，~150 token/次），用自己的引擎做推荐（硬过滤+评分排序）。这个决策的核心是'AI 做语言，算法做逻辑'，是产品决策而非技术决策——你要想清楚 AI 在产品里扮演什么角色。"

---

**Q3：如果你有 1000 个真实用户，你会怎么做？**

> "三件事：
>
> 第一，建立 Ground Truth 数据集。让用户对推荐打分（👍👎），收集 500 条有标注数据，衡量推荐质量的真实水平，找出哪类查询的成功率最低（比如敏感肌用户）。
>
> 第二，用户分群分析。看 Layer 1 vs Layer 2 用户的收藏率是否有差异，验证 Layer 1 的精准推荐是否带来更高的接受率。
>
> 第三，A/B 测试追问时机。当前策略是信息不足时追问，测试'直接给兜底推荐 + 提示可以细化'是否比追问带来更好的转化，因为用户可能因追问摩擦直接离开。"

---

**Q4：你的产品有什么缺点？**

> "三个主要缺点：
>
> **第一，跨会话无记忆**：用户每次对话是独立的，上次说了油皮，下次还要重新说。可以用 userId + 本地存储来解决，但涉及隐私设计。
>
> **第二，Layer 1 覆盖维度有限**：vegan 和 cruelty-free 关键词目前不进 Layer 1 检测，这类查询会走 Layer 2（慢 30 倍）。这是可以快速修复的低成本改进。
>
> **第三，产品库静态**：60 款固定产品，没有更新下架机制。商业化需要解决数据来源和实时更新问题。"

---

**Q5：如何判断这个产品是否成功？**

> "两层指标：
>
> **系统层**（现在能测的）：推荐准确率（硬条件 100% 符合）、Layer 1 命中率 50%、P95 响应时间 2.6s。
>
> **用户层**（有真实用户后）：推荐接受率（收藏率）、7 日回访率、平均对话轮数。
>
> North Star Metric 是**推荐接受率**，因为它最直接反映'推荐是否真正帮到了用户'。现在的系统层指标只是代理，用户层才是真正的价值验证。"

---

### 📌 AI 产品思维类问题

**Q6：你怎么看 AI 在产品中的角色？**

> "AI 是工具，不是目标。很多产品'为了 AI 而 AI'，用复杂方案解决了本来可以简单解决的问题。
>
> 在 Shine Beauty 里，AI 只做一件事：把用户的自然语言翻译成结构化意图。其他所有逻辑——过滤、评分、排序——都是确定性规则。好处是：AI 失效时（API 超时），系统自动降级到关键词层，用户体验不崩溃。
>
> Klook 的 AI 产品可能也是类似的思路——AI 增强体验，但不能成为单点故障。"

---

**Q7：如何处理 AI 推荐的不可解释性？**

> "在两个层面解决了这个问题：
>
> **技术层**：每条推荐附带 reason 字段（'适合油皮；哑光效果；$29 在$30预算内'），用户能看到推荐依据，不是黑盒。reason 字段是引擎自己生成的，不是 LLM 编的。
>
> **产品层**：推荐是建议，不是命令。话术是'这是我的推荐，告诉我是否想调整方向'，给用户控制权。透明度 + 控制权是建立用户对 AI 信任的关键。"

---

**Q8：AI 产品如何做 A/B 测试？**

> "AI 产品的 A/B 测试有两个特殊性：
>
> **指标滞后**：推荐好坏可能几天后才能反馈（用了觉得不好），不像 CTR 测试即时可得。需要设计好 Proxy Metric，比如推荐后 2 分钟内的收藏行为作为代理。
>
> **交叉效应**：A 组用关键词算法、B 组用 LLM，用户体验差异过大，很难控制其他变量。更好的做法是测单一变量，比如测追问阈值。
>
> 我的设计：对照组当前逻辑，实验组把 vegan/cruelty-free 加入 Layer 1 检测。主要指标：Layer 1 命中率变化；次要指标：这类查询的平均响应时间。"

---

### 📌 Klook 相关/通用 PM 问题

**Q9：你了解 Klook 吗？你为什么想来 Klook？**

> "Klook 是亚太最大的旅游体验预订平台，核心价值主张是帮用户发现和预订当地体验，把旅游从'景点打卡'变成'深度体验'。
>
> 我想来 Klook 有三个原因：第一，Klook 是 AI 推荐驱动的平台，个性化是核心竞争力，我在 Shine Beauty 里实践的推荐架构思维在这里有更大的舞台；第二，Klook 的用户跨语言跨文化，我处理中英双语用户的经验和业务场景高度匹配；第三，我想在高速成长的平台上学习如何做全球化产品。"

**Q10：如果你是 Klook 的 PM，你会如何改进搜索推荐？**

> "先框架再建议。
>
> **定义问题**：Klook 的搜索是'目的地意图'的入口，用户输入'东京'不是要找一个地点，而是要找'在东京能做什么有趣的事'。当前搜索偏向关键词匹配，没有充分利用用户意图。
>
> **机会点**：借鉴 Shine Beauty 的架构——用户输入'东京亲子游三天'，系统提取 {目的地: 东京, 人群: 亲子, 时长: 3天}，用结构化意图做个性化排序。
>
> **验证方式**：A/B 测试意图理解搜索 vs 传统关键词搜索，核心指标：搜索后的活动点击率和订单转化率。"

**Q11：如何在没有数据的情况下做产品决策？**

> "Shine Beauty 里面对的核心挑战——没有真实用户，所有决策基于假设。
>
> 三个方法：
>
> **用模拟测试替代用户数据**：设计代表真实用户行为的场景，用系统表现代理用户满意度。这不是完美方案，但是最小成本的 Proxy Validation。
>
> **用竞品分析补充认知**：分析 Sephora、Douglas 等平台的推荐逻辑，推断用户偏好和行为模式。
>
> **小范围定性访谈**：把产品给 3-5 个朋友用，观察行为而不只听反馈（行为 > 态度）。
>
> 结论：数据不足时，用最小成本获得最大信号，而不是等待完美数据。"

---

**Q12：说说你遇到的一个产品 tradeoff**

> "最典型的是追问频率：Precision（精准推荐）vs Friction（用户摩擦）。
>
> 追问越多推荐越准，但追问越多用户越可能放弃对话。
>
> 我的原则：只在信息缺失会导致推荐完全不准的时候才追问，不为追求完美而追问。具体：有效意图字段 ≥2 就直接推，否则精准追问缺失的那个字段。
>
> 结果：油皮+哑光这样的明确需求直接推（体验好），'推荐个粉底'这种完全模糊的需求追问（避免无效推荐）。"

---

**Q13：产品如何平衡用户需求和商业目标？**

> "在 Shine Beauty 里，用户目标是'找到真正适合我的粉底'，商业目标（如果商业化）是'促成购买'。短期可能冲突——用户$20预算，商业上更想推$60的产品。
>
> 我的立场：长期两者一致。真正帮用户找到好产品，建立信任，带来复购和口碑。牺牲用户体验来短期提升客单价，会摧毁推荐系统的信任基础。
>
> 我的设计原则：推荐必须首先满足用户明确表达的硬条件（预算、肤质），在约束范围内再考虑商业因素。"

> - **设计师**：用户流程、视觉一致性

## 十一、关键知识点速查

### 推荐系统基础
- **Collaborative Filtering**：基于用户行为相似性推荐（Shine Beauty 没用，因为没有用户数据）
- **Content-Based Filtering**：基于物品属性匹配（Shine Beauty 用的是这个：肤质、妆效、价格等属性）
- **Hard Filter**：必须满足的约束，不满足直接排除
- **Soft Scoring**：满足加分，不满足不扣分

### AI 产品常见术语
- **NLU**（Natural Language Understanding）：理解用户说了什么 → Shine Beauty L2 层做的事
- **Hallucination**：LLM 编造不存在的信息 → 用架构设计封死（LLM 不接触产品数据）
- **Prompt Engineering**：设计给 LLM 的指令 → 意图提取 prompt，temperature=0 确保确定性
- **Response Format**：`{ type: "json_object" }` → 强制 LLM 输出结构化 JSON
- **Fallback**：主方案失败时的降级策略 → L3 兜底层，永不崩溃
- **Enum Validation**：枚举白名单校验 → 防止 LLM 非法输出进数据库查询

### PM 框架速查
- **HEART**：Happiness / Engagement / Adoption / Retention / Task Success
- **RICE**：Reach / Impact / Confidence / Effort
- **North Star Metric**：推荐接受率（最直接反映用户价值）
- **Proxy Metric**：在没有直接数据时，用代理指标替代（收藏率代理购买意图）
- **Product-Market Fit**：用户愿意持续使用且推荐给他人
